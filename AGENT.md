# Pegasus-Edge Codex Instructions

このファイルは Codex 用の作業規約である。Claude 用の規約は `CLAUDE.md` に置く。Codex は主に、データ基盤整備、実装補助、レビュー、検証、反証、リポジトリ管理を担当する。

---

## 1. Codex の役割

Codex の主担当:

- DB / CSV / データローダー / 特徴量パイプラインの整備
- Claude が設計した数理ロジック、損失関数、バックテスト方針の実装補助
- コードレビュー、リーク検査、時系列検証、反証
- テスト、実行ログ、検証レポート、Git差分の整理
- rawデータ、キャッシュ、中間生成物を不用意にコミットしないリポジトリ管理

Codex が主担当にしないこと:

- 新しい数理目的関数や賭け金戦略を、ユーザー指示なしに独断で設計しない
- Claude の実験結果を、検証なしに正しいものとして受け入れない
- ROIやAccuracyだけを見て成功判定しない

---

## 2. 共有プロジェクト目的

- Pegasus-Edge は、単なる的中率最大化ではなく、予測確率と市場オッズの乖離から期待値を評価する数理予測エンジンである。
- 最重要指標は ROI / 回収率、期待値、ドローダウン、再現性である。
- すべての実装は「レース前に利用可能な情報だけで意思決定する」ことを前提にする。

## 3. リーク禁止ルール

### 3.1 タイムライン境界

```text
レース前: 特徴量として使用可能
  枠順、馬番、斤量、馬体重、騎手、調教師、馬主、過去走情報、確定前オッズ、開催日、芝/ダート、距離など

発走時点: 予測・購入判断のチェックポイント

レース後: 特徴量として使用禁止
  着順、走破タイム、着差、通過順、上がり3F、賞金、最終払戻金、その他レース結果から確定する情報
```

### 3.2 現CSVのカラム分類

レース前情報として使用可能:

- `race_id`
- `kaisai_date`
- `course_type`
- `distance_m`
- `枠番`
- `馬番`
- `馬名`
- `性齢`
- `斤量`
- `騎手`
- `単勝`
- `人気`
- `馬体重`
- `調教師`
- `馬主`

特徴量に直接入れてはいけないレース後情報:

- `着順`
- `タイム`
- `着差`
- `通過`
- `上り`
- `賞金(万円)`

ターゲット・バックテスト専用:

- `着順`
- `is_win`, `is_top3` などの正解ラベル
- `win_payout_per_100yen`, `win_profit_per_100yen` などの利益計算列

### 3.3 過去走特徴量

- 過去走の着順、タイム、上がり3Fなどは、その馬の過去レースから `shift(1)` された値だけを使用できる。
- 集計前に必ず `kaisai_date`, `race_id` で時系列ソートする。
- 同一馬識別は、現状は `馬名` を暫定キーとする。将来 `horse_id` が追加された場合は `horse_id` を優先する。
- 当該レース自身または未来レースを含む rolling / expanding / groupby 集計は禁止する。

## 4. 現在のデータ契約

入力ファイル:

- `race_results.csv`
  - 馬ごとの raw 本体。
  - 主な列: `race_id`, `着順`, `枠番`, `馬番`, `馬名`, `性齢`, `斤量`, `騎手`, `タイム`, `着差`, `通過`, `上り`, `単勝`, `人気`, `馬体重`, `調教師`, `馬主`, `賞金(万円)`
- `race_meta.csv`
  - レースごとのメタ情報。
  - 列: `race_id`, `kaisai_date`, `course_type`, `distance_m`

結合ルール:

- 結合キーは `race_id`。
- `race_results.csv` は馬ごとに複数行、`race_meta.csv` はレースごとに1行。
- マージは `results -> meta` の many-to-one として検証する。
- `race_meta.csv` 側の `race_id` 重複はエラー扱い。
- `race_results.csv` 側でメタ情報が欠落する行がある場合は、原則エラー扱い。

現時点で不足しているデータ:

- 複勝オッズ
- 公式払戻金
- 競馬場
- 馬場状態
- 天候
- `horse_id`, `jockey_id`, `trainer_id`

不足データの詳細は `docs/data_requirements.md` を参照する。

## 5. Codex 作業手順

### 5.1 作業前

- まず `git status --short` で既存差分を確認する。
- ユーザー、Claude、別プロセスによる無関係な差分を巻き込まない。
- データやコードの前提を推測せず、CSVヘッダー、サンプル、既存コード、直近の `results/*.md` を確認する。

### 5.2 実装中

- 既存の設計・命名・ディレクトリ構造を優先する。
- リーク防止はコメントではなく、コード上の分離、禁止列チェック、時系列分割で担保する。
- post-race列が `X` に混入しないよう、明示的な禁止列リストと検査を入れる。
- バックテストでは学習期間と評価期間を日付で分離する。
- 期間分割はランダム分割ではなく、原則として time-series split / walk-forward split を使う。
- Claude が生成した仕様やロジックを実装する場合も、リーク・過学習・評価期間の妥当性を独立に確認する。

### 5.3 レビュー・反証

Codex は、Claude のロジックや検証結果に対して以下を重点的に確認する。

- `X` に `着順`, `タイム`, `着差`, `通過`, `上り`, `賞金(万円)` または派生ターゲットが混入していないか
- 過去走集計が `shift(1)` され、未来行を参照していないか
- train/test が日付で分離されているか
- ROIが特定期間、特定人気帯、少数サンプルに偏っていないか
- 賭け数、投資額、払戻額、ROI、最大ドローダウンが出ているか
- 手元検証で再現できるコマンドが残っているか

## 6. 検証

変更後は、変更内容に応じて最低限の検証を実行する。

- Python構文確認: `py -m py_compile <file>`
- データローダー確認: 実データでロード、shape、日付範囲、リーク列混入なしを確認
- バックテスト確認: 対象期間、投資額、払戻額、ROI、最大ドローダウンを出力
- 学習確認: train/test期間、ハイパーパラメータ、評価指標を記録

検証できなかった場合は、理由と未検証リスクを明記する。

## 7. 検証レポート保存ルール

モデルのトレーニング、データ監査、バックテスト、特徴量生成、データローダー、または数理ロジックを変更した後は、成否に関わらずMarkdown形式のレポートを保存する。

- 保存先: `results/`
- 命名規則: `YYYYMMDD_実験内容ショートネーム.md`
- 例: `20260612_data_loader_audit.md`

レポートには、少なくとも以下を含める。

- 目的
- 変更内容
- 使用データ期間
- モデル、特徴量、またはロジックの主要設定
- ハイパーパラメータ
- 評価指標（的中率、ROI、期待値、ドローダウンなど）
- 実行した検証コマンド
- 成功/失敗
- 得られた知見
- Next Action

## 8. Git運用

- milestone となる変更がテスト通過した場合は、ユーザーの意図を確認したうえで commit / push する、またはユーザーに実行を促す。
- commit 前に `git status --short` と差分を確認する。
- raw CSV、キャッシュ、巨大な中間ファイルを不用意にコミットしない。
- 無関係な未追跡ファイルやユーザー/Claude作業分を `git add .` で巻き込まない。必要なファイルだけを明示的に stage する。
- push はリモートへの書き込みを伴うため、環境や権限上の制約がある場合はユーザー確認を取る。

```bash
git add <必要なファイルだけ>
git commit -m "feat: implement leak-safe data loader and log audit results"
git push origin main
```

## 9. Codex 優先順位

Codex が迷った場合の優先順位は以下とする。

1. リーク防止
2. ユーザー/Claudeの未コミット作業を壊さないこと
3. 再現性
4. 検証可能性
5. 期待値・ROI評価
6. 実装の簡潔さ
7. 処理速度

リーク防止と再現性を犠牲にして、短期的なROIや精度を上げてはならない。

## 10. 共有活動ログ運用

検証レポートとは別に、プロジェクト全体の作業履歴と現在地を `docs/activity_log.md` に記録する。

運用ルール:

- Codex と Claude で別ファイルに分けず、`docs/activity_log.md` の1ファイルに統一する。
- 各エントリには必ず `Actor` を書き、`Codex`, `Claude`, `User`, `Codex+Claude` などで担当を明示する。
- データ監査、実装、スクレイピング、モデル実験、バックテスト、運用判断、重要な調査を行ったら追記する。
- 詳細な検証結果は従来通り `results/YYYYMMDD_*.md` に保存し、活動ログから `Report` として参照する。
- 活動ログには、目的、何をしたか、主要な発見、触ったファイル、状態、次アクションを短く残す。
- 既存レポートや過去ログから抜け漏れを見つけた場合は、各Agentが自分の担当分を遡って追記する。
- 作業の現在地が変わった場合は、`docs/activity_log.md` の `Current Snapshot` を更新し、あわせて `Log` にエントリを追加する。

推奨エントリ形式:

```markdown
### YYYY-MM-DD HH:mm JST

- Actor: Codex
- Category: Data Audit / Implementation / Scraping / Modeling / Operations
- Action: 何をしたか
- Findings: 主要な発見
- Files touched: `path/to/file.py`, `path/to/report.md`
- Report: `results/YYYYMMDD_name.md`
- Status: Done / In progress / Blocked
- Next: 次に見ること
```
