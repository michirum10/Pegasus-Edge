# Pegasus-Edge Claude Instructions

このファイルは Claude 用の作業規約である。Codex 用の規約は `AGENT.md` に置く。Claude は主に、数理ロジック、特徴量設計、学習方針、バックテスト設計、実験検証を担当する。

---

## 1. Claude の役割

Claude の主担当:

- 期待値エッジを狙う数理ロジックの設計
- 損失関数、目的関数、賭け金配分、見送り条件の設計
- 特徴量仮説の立案と実験計画
- バックテスト設計、評価指標設計、結果分析
- 実験レポートの作成

Claude が特に重視すること:

- Accuracy ではなく、予測確率と市場オッズの乖離から期待値を評価する
- ROI、投資額、払戻額、賭け数、最大ドローダウン、年別/条件別の安定性を見る
- リーク、過学習、少数サンプルの偶然、人気帯の偏りを疑う
- Codex に実装・レビュー・反証を依頼しやすい形で仕様を残す

Claude が避けること:

- リーク検査なしにROI改善を成功扱いしない
- 学習期間の成績だけで結論を出さない
- データに存在しない列を前提にした実装をしない
- raw CSVを上書きしない

---

## 2. 共有プロジェクト目的

- Pegasus-Edge は、単なる的中率最大化ではなく、予測確率と市場オッズの乖離から期待値を評価する数理予測エンジンである。
- 最重要指標は ROI / 回収率、期待値、ドローダウン、再現性である。
- すべての実装は「レース前に利用可能な情報だけで意思決定する」ことを前提にする。

## 3. リーク禁止ルール

### 3.1 タイムライン境界

```text
レース前: 特徴量として使用可能
  枠順、馬番、斤量、馬体重、騎手、調教師、馬主、過去走情報、確定前オッズ、開催日、芝/ダート、距離など

発走時点: 予測・購入判断のチェックポイント

レース後: 特徴量として使用禁止
  着順、走破タイム、着差、通過順、上がり3F、賞金、最終払戻金、その他レース結果から確定する情報
```

### 3.2 現CSVのカラム分類

レース前情報として使用可能:

- `race_id`
- `kaisai_date`
- `course_type`
- `distance_m`
- `枠番`
- `馬番`
- `馬名`
- `性齢`
- `斤量`
- `騎手`
- `単勝`
- `人気`
- `馬体重`
- `調教師`
- `馬主`

特徴量に直接入れてはいけないレース後情報:

- `着順`
- `タイム`
- `着差`
- `通過`
- `上り`
- `賞金(万円)`

ターゲット・バックテスト専用:

- `着順`
- `is_win`, `is_top3` などの正解ラベル
- `win_payout_per_100yen`, `win_profit_per_100yen` などの利益計算列

### 3.3 過去走特徴量

- 過去走の着順、タイム、上がり3Fなどは、その馬の過去レースから `shift(1)` された値だけを使用できる。
- 集計前に必ず `kaisai_date`, `race_id` で時系列ソートする。
- 同一馬識別は、現状は `馬名` を暫定キーとする。将来 `horse_id` が追加された場合は `horse_id` を優先する。
- 当該レース自身または未来レースを含む rolling / expanding / groupby 集計は禁止する。

## 4. 現在のデータ契約

入力ファイル:

- `race_results.csv`
  - 馬ごとの raw 本体。
  - 主な列: `race_id`, `着順`, `枠番`, `馬番`, `馬名`, `性齢`, `斤量`, `騎手`, `タイム`, `着差`, `通過`, `上り`, `単勝`, `人気`, `馬体重`, `調教師`, `馬主`, `賞金(万円)`
- `race_meta.csv`
  - レースごとのメタ情報。
  - 列: `race_id`, `kaisai_date`, `course_type`, `distance_m`

結合ルール:

- 結合キーは `race_id`。
- `race_results.csv` は馬ごとに複数行、`race_meta.csv` はレースごとに1行。
- マージは `results -> meta` の many-to-one として検証する。
- `race_meta.csv` 側の `race_id` 重複はエラー扱い。
- `race_results.csv` 側でメタ情報が欠落する行がある場合は、原則エラー扱い。

現時点で不足しているデータ:

- 複勝オッズ
- 公式払戻金
- 競馬場
- 馬場状態
- 天候
- `horse_id`, `jockey_id`, `trainer_id`

不足データの詳細は `docs/data_requirements.md` を参照する。

## 5. Claude 作業手順

### 5.1 実験前

- まず `git status --short` を確認し、Codexやユーザーの未コミット作業を把握する。
- 直近の `results/*.md` を読み、既に失敗した仮説や既存の検証条件を確認する。
- 実験の目的、仮説、評価期間、成功条件、失敗条件を先に明文化する。
- データに存在しない列を使う場合は、先に `docs/data_requirements.md` を更新し、取得方針を決める。

### 5.2 ロジック設計

Claude は設計時に以下を明記する。

- 目的関数または損失関数の定義
- 予測確率と市場確率の関係
- 購入条件、見送り条件、賭け金配分
- 使用する特徴量と、各特徴量がレース前情報かどうか
- train/test期間
- walk-forward または time-series split の方法
- 評価指標と閾値

### 5.3 実験・検証

- 学習期間と評価期間を日付で分離する。
- ランダム分割だけで結論を出さない。
- 年別、course_type別、距離帯別、人気帯別などの分解評価を可能な範囲で行う。
- ROIが高い場合ほど、賭け数、最大ドローダウン、少数サンプル偏りを確認する。
- オッズを特徴量に使う場合、最終オッズなのか締切前オッズなのかを必ず明記する。

## 7. 検証レポート保存ルール

モデルのトレーニング、データ監査、バックテスト、特徴量生成、データローダー、または数理ロジックを変更した後は、成否に関わらずMarkdown形式のレポートを保存する。

- 保存先: `results/`
- 命名規則: `YYYYMMDD_実験内容ショートネーム.md`
- 例: `20260612_loss_function_v2_backtest.md`

レポートには、少なくとも以下を含める。

- 目的
- 仮説
- 変更内容
- 使用データ期間
- モデル、特徴量、ロジックの主要設定
- ハイパーパラメータ
- 評価指標（的中率、ROI、期待値、ドローダウンなど）
- 年別/条件別の分解
- 実行した検証コマンド
- 成功/失敗
- 得られた知見
- Next Action
- Codexにレビューしてほしい点

## 8. Git運用

- milestone となる変更がテスト通過した場合は、ユーザーの意図を確認したうえで commit / push する、またはユーザーに実行を促す。
- commit 前に `git status --short` と差分を確認する。
- raw CSV、キャッシュ、巨大な中間ファイルを不用意にコミットしない。
- Codexやユーザー作業分を `git add .` で巻き込まない。必要なファイルだけを明示的に stage する。

```bash
git add <必要なファイルだけ>
git commit -m "feat: evaluate value-based betting objective"
git push origin main
```

## 9. Claude 優先順位

Claude が迷った場合の優先順位は以下とする。

1. リーク防止
2. 実験仮説の明確さ
3. 再現性
4. 期待値・ROI評価
5. ドローダウンと安定性評価
6. Codexがレビュー可能な仕様化
7. 実装速度

リーク防止と再現性を犠牲にして、短期的なROIや精度を上げてはならない。
